# Requirements - 需求收集

**工作流阶段：需求收集**

基于功能创意生成 EARS 格式的初始需求集，然后与用户迭代完善，直到需求完整准确。

## 核心原则

### **CRITICAL：文件创建与格式要求**

- **MUST** 在 `.specs/requirements.md` 文件不存在时创建它
- **MUST** 根据用户的粗略想法生成需求文档的初始版本，无需先提出连续的问题
- **MUST** 按以下格式编排初始 requirements.md 文档：
  - 清晰的介绍部分，总结功能特性
  - 层次化的编号需求列表，每项包含：
    - 用户故事，格式为"作为[角色]，我希望[功能]，以便[收益]"
    - EARS 格式（易于理解的需求语法）的验收标准编号列表

### **示例格式**

```markdown
## 1. 用户认证功能

**用户故事**：作为系统用户，我希望能够安全登录系统，以便访问个人数据和功能。

**验收标准**：
1.1 当用户输入有效的用户名和密码时，系统应该允许用户登录
1.2 当用户输入无效凭据时，系统应该显示错误消息
1.3 当用户连续3次登录失败时，系统应该锁定账户15分钟
```

## 质量要求

### **IMPORTANT：需求质量要求**

- **MUST** 在初始需求中考虑核心功能场景和用户体验
- **SHOULD** 识别需要澄清的关键业务逻辑
- **MAY** 针对不明确的功能点提出确认问题
- **MUST** 确保需求可直接指导技术设计

## 流程控制

### **CRITICAL：阶段结束控制**

- **MUST** 生成需求文档后询问用户："需求文档是否准确完整？是否需要修改？"
- **MUST** 等待用户明确确认需求文档无误
- **NEVER** 在用户未明确确认需求文档前进入设计阶段
- **MUST** 如果用户要求修改，先完成修改再重新确认

## 语言规范

- **重要**：所有对话、注释、文档和输出都必须使用简体中文
- 保持简洁直接的沟通风格，避免冗余描述
- 直接给出解决方案，除非用户明确要求解释
